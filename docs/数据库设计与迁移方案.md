# 数据库设计与迁移方案

## 1. 当前存储现状

当前项目还没有切到正式关系型数据库，运行时主要采用：

- JSON 文件存储业务数据
- Neo4j 存储课程知识图谱/课程树关系

### 1.1 当前 JSON 存储清单

| 路径 | 类型 | 当前作用 |
|---|---|---|
| `data/Users/student.json` | JSON | 学生基本信息、学习目标、偏好、进度 |
| `data/Users/teacher.json` | JSON | 教师信息、学生绑定关系 |
| `data/digital_twins/*.json` | JSON | 学生数字孪生画像快照 |
| `data/learning_plans/*.json` | JSON | 学习路径、学习计划 |
| `data/Log/llm_log.json` | JSON | LLM 调用日志 |
| `data/sessions/*.json` | JSON | 会话状态、停留时长、当前节点 |
| `data/course/*.json` | JSON | 课程树、知识图谱原始数据 |

### 1.2 当前结论

当前系统属于：

- 存储结构已存在
- 业务可运行
- 但没有正式数据库 schema

因此本方案的目标是：

1. 先公开数据库设计
2. 后续再决定是否正式迁移

---

## 2. 建议的关系型数据库总体设计

建议未来统一采用：

- 关系型数据库：MySQL 或 PostgreSQL
- 图结构仍可保留 Neo4j，用于课程知识图谱关系

### 2.1 分层建议

| 层 | 建议技术 | 作用 |
|---|---|---|
| 用户与业务主数据 | MySQL / PostgreSQL | 用户、计划、日志索引、数字孪生快照 |
| 图关系数据 | Neo4j | 课程知识点关系、依赖路径、资源挂接 |
| 文件对象存储 | 本地磁盘 / OSS | PDF、视频、附件资源 |

## 5. 推荐迁移顺序

建议分 3 步迁移：

### 第一步：先保留 JSON，补数据库表

- 新建数据库
- 建表
- 不改现有功能
- 先让数据库结构稳定下来

### 第二步：双写

- 关键数据同时写 JSON 和数据库
- 先覆盖：
  - 用户信息
  - 会话信息
  - LLM 日志
  - 学生孪生快照

### 第三步：读数据库、保留 JSON 兜底

- 查询优先读数据库
- 老 JSON 作为迁移兜底
- 完成稳定后再考虑清理旧 JSON

---

## 6. 当前建议

当前最合理的状态不是立刻全量迁库，而是：

1. 先把数据库设计方案作为正式文档对外公开
2. 确认项目组是否真的要迁数据库
3. 如果确定迁移，再先做“用户 + 日志 + 学生孪生”三块

这样风险最小，也最符合当前项目成熟度。
