# 学生数字孪生总体设计与对接文档

## 1. 有哪些功能

本模块目标：基于学生在课程中的学习行为与结果，生成可用于“学生端展示 + 教师决策 + 课程孪生联动”的统一画像。

### 1.1 学生画像生成
- 输入：学生标识、课程标识、学习过程数据、测验数据、节点掌握数据
- 输出：课程维度学生画像（JSON）
- 核心指标：
  - `overall_mastery`：整体掌握度
  - `technical_level`：能力分层（基础薄弱/基础建立中/能力成型/进阶提升）
  - `radar`：知识掌握、学习投入、实践能力、学习稳定性、测验表现
  - `risk_alerts`：风险告警（知识薄弱、进度滞后、投入不足、趋势下滑等）
  - `trend`：趋势序列（按日/按采样点）

### 1.2 学生-课程画像查询能力
- API：`POST /api/digital-twin/student-course-profile`
- 入参：`student_id`、`course_id`
- 出参：该生该课完整画像 + 节点匹配结果 + 对其他孪生的输出字段

### 1.3 对其他孪生输出能力（跨模块协同）
- 输出给课程孪生：`outputs.for_course_twin`
  - 课程总体掌握度、风险等级、薄弱节点摘要
- 输出给教师孪生：`outputs.for_teacher_twin`
  - 技术等级、风险列表、趋势状态、薄弱节点摘要

### 1.4 双调用模式
- HTTP 模式（前后端/跨服务）  
  调用数字孪生 API 获取 JSON。
- 函数模式（后端内部）  
  `DigitalTwinModule.student_course_profile_service.get_student_course_profile(student_id, course_id)`  
  返回 Python `dict`（可直接序列化为 JSON）。

### 1.5 数据更新与回写
- 测验完成后更新节点状态与测验记录（`quiz_attempts`）
- 定时/触发式采集后更新 `twin_profiles` 与 `twin_profile_nodes`
- 画像读取优先实体表，保证可查询、可审计、可扩展

## 2. 指标计算公式（当前实现口径）

以下公式与当前后端实现一致，可作为评审与联调统一口径。

### 2.1 整体掌握度 `overall_mastery`

先计算每个知识点的 `mastery_score`：

`mastery_score = 0.40*quiz_score + 0.30*progress + 0.20*min(llm_interaction_count/10,1)*100 + 0.10*min(study_duration_minutes/30,1)*100`

规则：
- 当 `quiz_score >= 100` 且 `progress >= 100` 时，`mastery_score = max(mastery_score, 90)`
- `mastery_score` 结果裁剪到 `[0,100]`，保留 2 位小数

再计算整体掌握度：

`overall_mastery = average(mastery_score_i), i=1..N`

当 `N=0` 时返回 `0.0`。

### 2.2 能力分层 `technical_level`

设 `weak_count` 为薄弱节点数（`mastery_score < 60`）：

- `overall_mastery < 40`：基础薄弱
- `40 <= overall_mastery < 60`：基础建立中
- `60 <= overall_mastery < 80`：
  - `weak_count <= 4`：能力成型
  - 否则：基础建立中
- `overall_mastery >= 80`：进阶提升

### 2.3 雷达指标 `radar`

1) 知识掌握  
`knowledge_mastery = overall_mastery`

2) 学习投入 `engagement`  
`engagement = 0.6*avg(min(llm_interaction_count/10,1)*100) + 0.4*avg(min(study_duration_minutes/30,1)*100)`

3) 实践能力 `practice`  
`practice = 0.4*avg(progress) + 0.35*avg(quiz_score) + 0.25*engagement`

4) 学习稳定性 `stability`  
当趋势点数 `< 3` 时，固定为 `70.0`。  
当趋势点数 `>= 3`：
`delta_j = abs(m_j - m_(j-1))`  
`volatility = avg(delta_j)`  
`stability = max(0, 100 - min(4*volatility, 100))`

5) 测验表现  
`test_performance = max(avg(quiz_score), 0.8*avg(progress))`

### 2.4 风险告警 `risk_alerts`

命中规则即生成告警，最终按级别排序（`high > medium > low`）：

- `overall_mastery < 45` → `knowledge_gap/high`
- `45 <= overall_mastery < 60` → `knowledge_gap/medium`
- `avg(progress) < 50` → `progress_lag/high`
- `50 <= avg(progress) < 70` → `progress_lag/medium`
- `engagement < 45` → `engagement_low/medium`
- `trend_status == downward` → `trend_down/medium`
- `weak_nodes_count >= 5` → `weak_nodes_cluster/medium`

### 2.5 趋势 `trend`

设时间序列首尾掌握度分别为 `m_start`、`m_end`：

`change = round(m_end - m_start, 2)`

趋势判断：
- `change >= 5`：`upward`
- `change <= -5`：`downward`
- 其他：`stable`

当趋势点数 `<2` 时：`trend_status=stable`，`change=0.0`。

## 3. 设计的数据库

数据库：SQLite（`data/app.db`）  
设计原则：实体化优先、主键清晰、可追踪、可扩展。

### 2.1 核心实体表

| 表名 | 作用 | 主键/关键字段 | 与学生数字孪生关系 |
|---|---|---|---|
| `users` | 用户主数据 | `user_id`（PK）, `user_type`, `username`, `login_id` | 提供学生身份映射与登录标识 |
| `courses` | 课程主数据与快照 | `course_id`（PK）, `course_name`, `payload_json` | 提供课程结构主载体 |
| `course_nodes` | 课程节点明细 | `course_id + node_id`（唯一） | 用于节点匹配、薄弱点定位 |
| `resources` | 节点资源明细 | `resource_id`（PK）, `course_id`, `node_id` | 辅助学习路径与资源推荐 |
| `quiz_attempts` | 测验记录 | `attempt_id`（PK）, `user_id`, `course_id`, `node_id`, `score` | 提供测验表现与趋势输入 |
| `twin_profiles` | 学生孪生主画像 | `id`（PK）, `user_id`, `username`, `overall_mastery`, `payload_json` | 孪生总览与快照 |
| `twin_profile_nodes` | 孪生节点明细 | `id`（PK）, `profile_id`, `node_id`, `mastery_score`, `progress`, `quiz_score` | 节点级画像计算主来源 |
| `teacher_student_links` | 师生关系 | `id`（PK）, `teacher_username`, `student_username` | 教师孪生聚合学生画像时使用 |

### 2.2 推荐索引/约束
- `users.login_id`：唯一索引（建议）
- `course_nodes(course_id, node_id)`：联合唯一
- `resources(course_id, node_id, resource_path)`：联合唯一
- `quiz_attempts(user_id, course_id, node_id, created_at)`：查询索引
- `twin_profile_nodes(profile_id, node_id)`：联合唯一或索引

### 2.3 数据流（简版）
1. 学生行为/测验写入实体表（课程、节点、测验）  
2. 采集/计算服务更新孪生主表与节点表  
3. 查询接口按 `student_id + course_id` 聚合并返回标准 JSON  
4. 输出子结构供课程孪生与教师孪生直接消费

## 4. 需要别人提供的数据（对接清单）

你的模块要稳定产出，必须依赖其他组提供结构化输入。建议按“来源系统-字段-频率-质量要求”对齐。

### 3.1 账号与身份组提供
- 学生身份映射：`student_id` ↔ `user_id/login_id/username`
- 教师身份映射与师生关系
- 用户状态（在读/停用）变更通知

### 3.2 课程与内容组提供
- 课程唯一标识：`course_id`
- 课程知识结构（章节/小节/知识点）
- 节点唯一标识：`node_id`（全局或课程内唯一）
- 节点与课程归属关系

### 3.3 学习行为组提供
- 学习进度：`progress`（节点级）
- 学习时长：`study_duration_minutes`
- 交互次数：`llm_interaction_count`
- 事件时间戳：`created_at/updated_at`

### 3.4 测评组提供
- 测验记录：`score`, `total`, `passed`
- 节点映射：测验对应的 `node_id`
- 测评时间序列（用于趋势）

### 3.5 教师业务组提供
- 风险干预反馈（是否干预、干预结果）
- 班级/课程组织关系（用于教师视角聚合）

### 3.6 数据质量要求（必须约定）
- 标识一致性：`student_id/course_id/node_id` 在所有系统一致
- 时间一致性：统一时区与时间格式（ISO8601）
- 缺失值语义：明确 `0`、`null`、空数组的业务含义
- 去重策略：重复事件去重规则（事件 ID 或幂等键）
- 回补机制：历史数据补录后的重算策略

## 5. 模块边界与职责（建议你对外强调）

### 4.1 你负责（学生数字孪生组）
- 画像指标定义与算法实现
- 画像查询接口与函数调用输出契约
- 风险与趋势判定逻辑
- 对课程孪生/教师孪生输出字段稳定性

### 4.2 其他组负责
- 业务原始数据生产与准确性
- 账号、课程、测验等主数据唯一 ID 管理
- 上游数据变更通知与版本管理

### 4.3 联调验收口径
- 同一 `student_id + course_id`，API 与函数输出一致
- 关键字段不缺失：`overall_mastery/radar/trend/outputs`
- 错误码清晰：参数错误 400、对象不存在 404、系统异常 500

## 6. 调用示例

### 5.1 API 调用
```bash
curl -X POST "http://localhost:8000/api/digital-twin/student-course-profile" \
  -H "Content-Type: application/json" \
  -d "{\"student_id\":\"stuliuyiming\",\"course_id\":\"Spark\"}"
```

### 5.2 函数调用
```python
from DigitalTwinModule.student_course_profile_service import get_student_course_profile

result = get_student_course_profile("stuliuyiming", "Spark")
print(result)  # dict / JSON-compatible
```
